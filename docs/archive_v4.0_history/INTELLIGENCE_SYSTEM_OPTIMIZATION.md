# 情报系统优化方案

## 1. 现状分析

### 当前问题
- 情报越来越多，token越来越长
- 超出模型上下文限制
- 检索效率降低

### 当前数据库结构
```sql
CREATE TABLE intelligence (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    symbol TEXT NOT NULL,           -- 股票代码
    content TEXT NOT NULL,          -- 情报内容
    source TEXT,                    -- 来源
    priority TEXT DEFAULT 'normal',  -- 优先级: high/normal/low
    marked_by_user BOOLEAN DEFAULT 0, -- 用户标记
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    expires_at TIMESTAMP,           -- 过期时间
    is_active BOOLEAN DEFAULT 1     -- 是否活跃
);
```

---

## 2. 优化方案

### 方案一：分层存储（推荐）

#### 分层逻辑
```
┌─────────────────┐
│   活跃情报层     │  最近6个月 + 高优先级(priority=high)  → 参与RAG检索
├─────────────────┤
│   历史归档层     │  6个月前 或 低优先级   → 压缩后归档，不参与检索
└─────────────────┘
```

#### 实现步骤
1. 新增字段：`is_archived` (BOOLEAN)
2. 新增字段：`summary` (TEXT) - 压缩后的摘要
3. 定时任务：每月1日检查，6个月前的自动归档

---

### 方案二：自动压缩

#### 压缩时机
- 归档时自动压缩
- 保留：日期 + 关键词 + 核心结论（约100字）

#### 压缩示例
```
原文：2026年3月14日，美国商务部撤回AI芯片出口规则。该规则草案原定于2月26日发布...（500字）

压缩后：2026-03-14 美国撤回AI芯片出口规则，利好芯片股（100字）
```

---

### 方案三：检索优化

#### 当前问题
- 全量检索 → token太长

#### 优化后查询
```sql
-- 只检索最近N条 + 高优先级
SELECT content, summary 
FROM intelligence 
WHERE symbol = '588200' 
  AND is_active = 1 
  AND is_archived = 0
  AND created_at > datetime('now', '-6 months')
ORDER BY 
  CASE priority WHEN 'high' THEN 1 ELSE 2 END,
  created_at DESC 
LIMIT 10;
```

## 6. 可借鉴的架构（参考MemOS）

### MemOS的核心设计
- 向量存储（Qdrant）
- LLM过滤相关性
- 分层存储（活跃/归档）
- 自动摘要

### 借鉴到情报系统
```
┌─────────────────────────────────────┐
│           输入情报                      │
├─────────────────────────────────────┤
│  1. 向量化 → Qdrant向量库            │
│  2. LLM判断相关性 → 过滤噪音        │
│  3. 自动摘要 → 关键信息提取         │
│  4. 分层存储 → 活跃/归档           │
├─────────────────────────────────────┤
│           检索时                      │
│  1. 向量相似度搜索                  │
│  2. LLM相关性二次过滤              │
│  3. 返回Top N条 + 摘要             │
└─────────────────────────────────────┘
```

### 可复用的能力
- DeepSeek Reason：自动摘要、相关性判断
- Qdrant向量库：语义搜索
- MemOS插件架构：直接集成

### 优点
1. 不只是关键词搜索，是语义理解
2. LLM自动过滤不相关信息
3. 自动生成摘要，减少token
4. 向量检索更精准

## 7. 股票/ETF 关键词配置

### 核心思路
每只ETF关心的情报类型不同，设置关键词后自动匹配

### ETF关键词配置示例
```json
{
  "588200": {
    "name": "科创芯片ETF",
    "keywords": ["芯片", "半导体", "AI芯片", "国产替代", "科创板", "集成电路", "光刻机"],
    "tags": ["科技", "芯片", "国产替代"]
  },
  "588710": {
    "name": "科创半导体设备ETF", 
    "keywords": ["半导体设备", "光刻机", "晶圆", "设备国产替代", "芯片制造"],
    "tags": ["半导体", "设备", "国产替代"]
  },
  "588000": {
    "name": "上证50ETF",
    "keywords": ["上证50", "大盘", "蓝筹", "权重股"],
    "tags": ["大盘", "蓝筹"]
  }
}
```

### 自动匹配逻辑
```
新情报入库时：
1. 提取情报关键词
2. 遍历所有ETF关键词配置
3. 匹配到的ETF自动关联
4. 关联成功后推送给对应股票群
```

### 好处
- 情报自动归类，不漏掉
- 一条情报可关联多只ETF
- 推送更精准

## 8. 智能关联（LLM判断）

### 用LLM做情报-ETF匹配

**关键词的问题：**
- 匹配不准
- 同义词难处理
- 语义理解差

**LLM方案：**
```python
# 输入情报内容 + ETF配置
# LLM输出：关联的ETF列表 + 置信度

例如：
情报: "美国商务部撤回AI芯片出口规则，对半导体设备构成利好"
LLM判断: 
- 588710 (科创半导体设备ETF): 95% - 直接利好
- 588200 (科创芯片ETF): 80% - 间接利好
```

### 实现流程
```
新情报入库 
    ↓
LLM分析 (DeepSeek Reason)
    ↓
输出关联ETF + 置信度
    ↓
自动关联 + 推送到对应群
```

### 优点
- 语义理解，不只是关键词匹配
- 可以理解"利好"、"利空"等情感
- 可以处理同义词、上下文
